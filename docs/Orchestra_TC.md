# Техническое задание

## Проект

**Orchestra**

Версия: 1.0 (MVP)

> Это исполняемое ТЗ для разработчика (включая кодера MiMo). Здесь — «что реализовать».
> Архитектурные детали — в [Architecture.md](Architecture.md); контракты компонентов — в отдельных Protocol-документах; идеология — в [Vision 2030.md](Vision%202030.md). Этот документ ссылается на них как на single source of truth и не дублирует.

---

# 1. Цель проекта

Разработать десктопное/web-приложение, позволяющее нескольким LLM совместно обсуждать один проект по методологии GSD (Goal → Spec → Develop).

Система должна организовывать **структурированный инженерный диалог** между несколькими специализированными агентами, а не обычный чат.

Главная задача приложения — получать более качественные архитектурные решения за счёт управляемой дискуссии между экспертными ролями.

---

# 2. Главная идея

Не существует «главного ИИ».

Каждый агент отвечает только за свою область компетенции. Все сообщения проходят через единый Orchestrator. В конце каждого раунда Consensus Engine формирует итоговое решение.

---

# 3. Основные роли

> Полное описание ролей, декларативных манифестов и контрактов — в [Agent Protocol.md](Agent%20Protocol.md).

| Роль | Назначение |
|---|---|
| **ChatGPT** — Chief Architect | Архитектура, системное проектирование, декомпозиция, стратегии, риски. Никогда не пишет код. |
| **GLM** — Tech Lead | Реализация, производительность, стек, инфраструктура, API, БД, DevOps. Без бизнес-логики. |

| **Critic** — Red Team | Поиск ошибок, угрозы, логические противоречия, нарушение SOLID/GSD/архитектурных принципов. |
| **MiMo** — Senior Software Engineer | Код, рефакторинг, миграции, тесты, исправление ошибок. Без самостоятельных архитектурных решений. |
| **Consensus Engine** (не LLM) | Сбор ответов, поиск согласия/противоречий, итог, следующий шаг GSD. |

# 4. Методология GSD

Каждая задача проходит одинаковый жизненный цикл.

```text
Discover → Goal → Specification → Architecture
        → Implementation → Review → Consensus → Iteration
```

Переход на следующую фазу невозможен до завершения текущей **и** до прохождения порога gating по Decision Confidence.

> Полный контракт фаз, артефактов и gating — в [GSD Integration.md](GSD%20Integration.md).

# 8. Conducting Score

**Требование к реализации (новый UI-паттерн):** первичный интерфейс — Conducting Score, замещающий обычный чат.

Свойства, обязательные для реализации:

- горизонтальное время (раунды слева направо);
- вертикальные независимые дорожки ролей;
- синхронные аккорды — на одном такте видно ответы всех ролей по теме;
- сила согласия кодируется визуально (Decision Confidence);
- клик по аккорду открывает Decision/ADR, а не лог чата;
- стриминг токенов подсвечивает активные дорожки в реальном времени;
- переход к обычным чат-пузырям **запрещён** каноном.

> Канон-инварианты и визуальные правила — в [UI Canon.md §1, §9](UI%20Canon.md).

# 9. Discussion Graph (функциональная возможность)

Обсуждение — это **граф параллельных веток**, а не линейный поток. Это функциональная возможность системы, а не только концепция.

```text
                Выбор БД
                     │
      ┌──────────────┼──────────────┐
 PostgreSQL      MongoDB      EventStore
      │              │              │
      └──────────────┼──────────────┘
                     │
             Consensus Engine
                     │
                     ▼
            Architecture Decision
```

**Требования к реализации:**

- возможность создать **ветку** из любого обсуждаемого утверждения;
- каждая ветка имеет собственный контекст, исследования, риски, Consensus и артефакты;
- ветки визуализируются как expand/collapse дерево поверх Conducting Score (см. [UI Canon.md §4](UI%20Canon.md));
- merge ветки в main graph возможен только после явного `Approve` дирижёра;
- сворачивание ветки не теряет её контекст (он сохранён в Knowledge Graph).

---

# 10. Discussion Branch Engine

**🧩 Движок параллельных веток обсуждения.** Архитектурно — подсистема Context Service, управляющая изоляцией контекста между ветками Discussion Graph.

**Контракт движка:**

```typescript
interface DiscussionBranchEngine {
  // Создать новую ветку из утверждения/узла
  forkBranch(parentNodeId: string, topic: string): BranchId;

  // Изолировать контекст ветки: отдельное подмножество Knowledge Graph
  isolateContext(branchId: BranchId): ContextScope;

  // Запустить независимый Consensus внутри ветки
  runBranchConsensus(branchId: BranchId): ConsensusReport;

  // Слить результаты ветки в main graph (требует Approve дирижёра)
  mergeBranch(branchId: BranchId, approver: 'conductor'): MergeResult;

  // Закрыть ветку без merge (отклонённая альтернатива)
  abandonBranch(branchId: BranchId): void;
}
```

**Требования:**

- контекст ветки изолирован — агенты в ветке не видят другие ветки без явного запроса;
- каждая ветка имеет собственный Consensus и собственные метрики Decision Confidence;
- gating применяется **внутри** ветки независимо;
- `mergeBranch` создаёт версионные отношения `replaces`/`supersedes` в Knowledge Graph;
- полный lifecycle ветки журналируется на Event Bus (см. [Architecture.md §5](Architecture.md)).

# 11. Decision Confidence (требование к реализации)

Каждое решение и каждый Consensus Report должны сопровождаться набором метрик в процентах:

```
Architecture Confidence
Implementation Confidence
Research Coverage
Risk Coverage
Test Coverage
overall
```

**Требования:**

- метрики пересчитываются после каждого ответа любой роли (**Continuous Consensus**, см. §12);
- метрики отображаются в шапке Conducting Score как gauge-индикаторы (см. [UI Canon.md §2](UI%20Canon.md));
- при падении метрики ниже порога gating соответствующий переход фазы GSD **блокируется**;
- показ одной сводной цифрой вместо пяти метрик **запрещён**.

> Определения метрик, формулы вычисления и пороги gating — в [Consensus Protocol.md §5, §6](Consensus%20Protocol.md).

---

# 12. Continuous Consensus (требование к реализации)

Consensus пересчитывается **постоянно**, после каждого ответа любой роли, а не только в фазе Consensus.

**Требования:**

- после каждого `AgentResponded` события Consensus Engine пересчитывает метрики и публикует `ConfidenceRecalculated`;
- UI обновляется в реальном времени (WebSocket), без перезагрузки;
- вновь пришедший ответ пересобирает «вертикаль» Conducting Score — дирижёр видит, как новое утверждение меняет уровень согласия;
- состояние «ожидание» подсвечивает дорожки, ответ которых повысит Confidence.

> Алгоритм и поток событий — в [Consensus Protocol.md §2, §4](Consensus%20Protocol.md).

# 14. Prompt Registry

Для каждой роли существует отдельный системный промпт.

```
/prompts
  architect.md
  tech_lead.md
  critic.md
  researcher.md
  engineer.md
```

Редактирование промптов производится **без перекомпиляции приложения**. Версия промпта фиксируется в каждом Context Packet для воспроизводимости.

# 19. Архитектурные принципы

1. GSD является обязательным жизненным циклом каждой задачи.
2. Все решения проходят через Consensus Engine.
3. Ни одна роль не может изменить решение другой роли напрямую.
4. Все изменения фиксируются как отдельные артефакты.
5. Каждое архитектурное решение имеет трассируемую историю.
6. Провайдеры LLM взаимозаменяемы благодаря адаптерной модели.
7. Приложение должно поддерживать расширение новыми ролями без изменения ядра.
8. Все обсуждения экспортируются в воспроизводимые инженерные артефакты (ADR, Markdown, JSON).
9. Прямой обмен полной историей сообщений между агентами **запрещён** — только через Context Packet.

> Полный свод принципов и контрактов расширения — в [Architecture.md §1, §11](Architecture.md) и [Agent Protocol.md §8](Agent%20Protocol.md) (Plugin SDK).

# 20. Человек как финальный арбитр

Orchestra не принимает необратимых решений самостоятельно.

Все архитектурные изменения, переходы между фазами GSD и запуск кодогенерации требуют подтверждения пользователя.

Роль человека — главный архитектор процесса и владелец решения (дирижёр).

> Распределение ответственности по фазам — в [GSD Integration.md §4, §5](GSD%20Integration.md). Концепция — в [Vision 2030.md §Человек как дирижёр](Vision%202030.md).

---

# 21. Главный архитектурный принцип

Orchestra — это не мультичат и не оболочка над несколькими LLM.

Это **платформа управления инженерным мышлением**, где:

* GSD определяет жизненный цикл разработки;
* Knowledge Graph хранит знания проекта;
* Context Service доставляет каждой роли только необходимую информацию;
* Role Router обеспечивает независимость агентов;
* Consensus Engine превращает экспертную дискуссию в формализованные инженерные решения;
* Decision Repository сохраняет результаты в виде воспроизводимых артефактов;
* Conducting Score делает всё это видимым для дирижёра как партитура, а не как чат.

Система должна проектироваться как расширяемая операционная среда для коллективной работы человека и специализированных ИИ-агентов, а не как очередной интерфейс для общения с моделями.

# Видение версии 2.0

После MVP система должна эволюционировать в полноценную платформу коллективной инженерии с поддержкой:

* автоматического запуска агентов по событию;
* параллельных обсуждений (Discussion Branch Engine — см. §10);
* голосования между ролями;
* оценки уверенности каждого агента (Decision Confidence — см. §11);
* автоматического построения архитектурных диаграмм;
* интеграции с GitHub, GitLab и Jira;
* автоматической генерации pull request по итогам Consensus Engine;
* поддержки локальных LLM и облачных моделей одновременно;
* режима полностью автономного GSD-цикла с участием человека только на контрольных точках.

> Долгосрочное направление — в [Vision 2030.md §Дорожная карта 2030+](Vision%202030.md).
