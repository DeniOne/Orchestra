# UI Canon

> Канон интерфейса Orchestra: Conducting Score как первичный UI-паттерн, layout, визуализация Decision Confidence и Continuous Consensus.
> Смежные: [Orchestra_TC.md §7–9](Orchestra_TC.md), [Consensus Protocol.md](Consensus%20Protocol.md) (метрики Confidence), [Architecture.md](Architecture.md).

Orchestra — **не** чат. Поэтому и UI — не окно сообщений, а **Conducting Score**: дирижёрская партитура, где дорожки — агенты, такты — раунды, аккорды — инженерные решения. Этот документ задаёт визуальный канон и запрещает деградацию к обычному мультичату.

---

## 1. Conducting Score — первичный UI-паттерн

Метафора: **дирижёр и партитура**. Пользователь — дирижёр, который не пишет каждую ноту за музыкантов, а управляет ансамблем: задаёт темп, вводит инструменты, утверждает аккорды.

### Аналогия

| Партитура | Orchestra |
|---|---|
| Дорожка (stave) | Роль-агент (ChatGPT, GLM, Gemini, Critic, MiMo) |
| Такт (measure) | Раунд GSD |
| Нота / аккорд | Утверждение / инженерное решение |
| Динамические знаки (f, p) | Decision Confidence (сила согласия) |
| Фермата (остановка) | Gating FAIL — переход заблокирован |
| Репетиционный знак | ADR / точка утверждения дирижёром |
| Ветвление (voltas) | Ветка Discussion Graph |

### Свойства Conducting Score

- **Горизонтальное время** — раунды идут слева направо, так же как такты в партитуре.
- **Вертикальные роли** — каждая дорожка независима (см. [Agent Protocol.md §Agent Independence](Agent%20Protocol.md#2-agent-independence)).
- **Синхронные аккорды** — на одном такте видно, что сказали все роли по одной теме; это и есть материал для Consensus.
- **Видимая сила согласия** — насыщенность/высота аккорда кодирует Decision Confidence.
- **Фокус — решение, а не сообщение** — клик по аккорду раскрывает Decision/ADR, а не лог чата.
- **Stream-friendly** — стриминг токенов подсвечивает активные дорожки в реальном времени.

## 2. Decision Confidence UI

Decision Confidence (см. [Consensus Protocol.md §5](Consensus%20Protocol.md#5-decision-confidence)) отображается как набор gauge-индикаторов в шапке Conducting Score.

```text
Architecture   ◐◐◐◐◐◐◐◐◐○  94 %
Implementation ◐◐◐◐◐◐◐◐○○  88 %
Research       ◐◐◐◐◐◐◐◐◐○  92 %
Risk           ◐◐◐◐◐◐◐○○○  79 %
Test           ◐◐◐◐◐◐○○○○  64 %    ⚠ below threshold
```

Правила отображения:
- **Зелёный** — метрика выше порога gating для текущей фазы.
- **Жёлтый** — близко к порогу (Δ ≤ 5 %).
- **Красный** — ниже порога; **переход фазы заблокирован**.
- Tooltip показывает вклад каждой роли и пробелы (недостающие исследования, непокрытые риски).

## 3. Continuous Consensus display

Consensus пересчитывается после каждого ответа (см. [Consensus Protocol.md §Continuous Consensus](Consensus%20Protocol.md#4-continuous-consensus)). UI отражает это в реальном времени:

- Панель **CONSENSUS** обновляется без перезагрузки (WebSocket-стрим `ConfidenceRecalculated`).
- На Conducting Score вновь пришедший аккорд **пересобирает вертикаль** — дирижёр видит, как новое утверждение меняет уровень согласия.
- Состояние «ожидание» (WAIT) — подсветка дорожек, ответ которых достаточен для повышения Confidence.

## 4. Discussion Graph в UI

Обсуждение — граф параллельных веток (см. [Architecture.md §Discussion Graph](Architecture.md#7-discussion-graph)). В UI он представлен как **expand/collapse дерево веток** поверх Conducting Score.

```text
Main │ ▶ Choose DB
     │   ├─▶ PostgreSQL   [Consensus 91 %]
     │   ├─▶ MongoDB      [Consensus 67 %]  ⚠ open risk
     │   └─▶ EventStore   [Consensus 58 %]  ⚠ needs research
```

- Каждая ветка — отдельный под-раунд с собственным контекстом и Consensus.
- Сворачивание ветки не теряет её контекст (он сохранён в Knowledge Graph).
- Merge ветки в main graph — только после явного `Approve` дирижёра.

## 6. Панель сообщений (дорожка роли)

Каждый блок дорожки отображает:

- роль;
- последний ответ;
- историю сообщений (по тактам);
- статус (active / waiting / done);
- время;
- количество токенов;
- стоимость запроса (если известна).

## 7. Панель управления

```
Start Session
Next Phase
Send
Approve
Reject
Request Revision
Generate Consensus
Export ADR
Export Markdown
Export PDF
Finish Session
```

`Next Phase` **отключён**, пока gating-вердикт `fail` (см. [Consensus Protocol.md §6](Consensus%20Protocol.md#6-gating-по-фазам-gsd)). Это визуальное отражение инварианта: фазовый переход невозможен без достаточной уверенности.

## 8. Нефункциональные требования к UI

- Полная поддержка потоковой генерации (streaming) — обязательно для живого Conducting Score.
- Возможность остановить ответ любого агента (`cancel`).
- Автосохранение сессий.
- Поддержка проектов с тысячами сообщений без деградации.
- Время открытия проекта менее 2 секунд.
- Полная локализация интерфейса на русский язык.
- Светлая тема по умолчанию.
- Все действия журналируются (события Event Bus).

---

## 9. Канон-инварианты (что запрещено)

1. **Нельзя** превращать дорожку роли в обычный чат-пузырь — нарушает метафору партитуры.
2. **Нельзя** показывать Confidence как один сводный процент — теряется диагностическая ценность (5 метрик).
3. **Нельзя** позволять переход фазы при красной метрике без явного owner-override с записью в аудит.
4. **Нельзя** показывать сырую историю сообщений там, где достаточно ADR/Decision-артефакта (см. [Context Protocol.md](Context%20Protocol.md)).
5. **Нельзя** давать агентам прямой обмен в UI — все взаимодействия идут через Conducting Score и Consensus.
