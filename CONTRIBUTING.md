# Contributing to Orchestra

Спасибо за интерес к Orchestra. Этот репозиторий сейчас находится в активной стадии проектирования: основная ценность — инженерная документация в [`docs/`](docs/). Контрибьюции в документацию так же важны, как и будущий код.

## С чего начать

1. Прочитайте [docs/Vision 2030.md](docs/Vision%202030.md) — идеология и направление.
2. Прочитайте [docs/Architecture.md](docs/Architecture.md) — как устроено.
3. Прочитайте [docs/Orchestra_TC.md](docs/Orchestra_TC.md) — что реализуется.

Это даёт общую картину: каждый документ отвечает на один вопрос, и они согласованы между собой.

## Соглашения по документации

- **Язык:** русский. Соответствует существующему стилю всех документов в `docs/`.
- **Формат:** GitHub-flavored Markdown. Используйте ASCII-диаграммы для потоков (как в [Architecture.md §4](docs/Architecture.md)) и Mermaid для C4/state/flowchart-диаграмм.
- **Структура секций:** нумерованные заголовки (`# 1. ...`, `# 2. ...`) — как в Protocol-документах.
- **Code blocks:** TypeScript для контрактов интерфейсов, YAML для декларативных манифестов (Role Manifest, контекстные политики).
- **Single source of truth:** каждое понятие (Knowledge Graph, Memory Layers, Event Bus, Consensus, Conducting Score) имеет **один** canonical-документ. Остальные документы ссылаются, а не дублируют. Перед добавлением раздела проверьте — не описано ли это уже в другом `docs/*.md`.

## Как добавлять ADR (Architecture Decision Records)

Когда принимается значимое архитектурное решение, оно фиксируется как ADR:

1. Создайте `docs/ADR/NNNN-короткое-название.md` (например, `0001-event-bus-choice.md`), где `NNNN` — следующий порядковый номер.
2. Структура ADR: **Context** (контекст) → **Decision** (решение) → **Consequences** (последствия).
3. ADR неизменяем после утверждения; новая версия — новый ADR с пометкой `supersedes ADR-NNNN`.

> Папка `docs/ADR/` появится с первым ADR. Пока что она упомянута в [README.md](README.md) как roadmap.

## Принципы изменений

- **Согласованность имён:** проект называется **Orchestra**. Не используйте устаревшее «AI Council Studio» в новых текстах.
- **Зависимости:** при изменении концепции, которая описана в нескольких документах, обновляйте **все** зависимые места. Недосинхрон канон-документов недопустим.
- **Честность:** README, TC и ADR обязаны отражать фактическое состояние. Никаких fake-green: «готово» пишется только когда действительно готово.
- **Разделение фаз GSD:** отдельное архитектурное изменение → отдельный ADR → отдельная фаза разработки. Не смешивайте.

## Стиль коммитов

Пишите содержательные сообщения, отражающие **что** и **почему** (не только «что»). Пример:

```
docs(architecture): добавить C4 Container-диаграмму и каталог событий Event Bus
```

## Связь с GSD-циклом

Orchestra реализует GSD как внутренний конечный автомат (см. [docs/GSD Integration.md §6](docs/GSD%20Integration.md)). Сама разработка Orchestra может вестись по внешнему GSD-циклу (PLAN → код → review → gate), но эти два цикла **не смешиваются**: внешний строит Orchestra как продукт, внутренний работает внутри Orchestra над решениями пользователя.

## Вопросы и обсуждения

Открывайте issue для предложений по архитектуре, обсуждения ADR или сообщения о рассинхроне между документами.
