## 0. Контекст

### 0.1. Долг D-10 — мёртвый manifest.model (КРИТИЧНЫЙ)

Разведка tech lead вскрыла **3 независимых несогласованных источника «model»** в кодовой базе:

**(1) `role-manifests/*.yaml` `model:` поле — ПОЛНОСТЬЮ МЁРТВОЕ.** Парсится в `RoleManifest.model`
(`packages/domain/src/agent.ts:8`), но **НИ ОДИН ридер не использует** (zero readers в репо). Каждый
манифест имеет `model:` (`architect.yaml: model: gpt-4o`, `engineer.yaml: model: mimo-7b`, и т.д.) —
но это dead data.

**(2) `packet-builder.ts:139-148` `resolveModelTarget()` — хардкод-таблица с НЕВАЛИДНЫМИ значениями:**
```typescript
function resolveModelTarget(roleId: string): string {
  const map: Record<string, string> = {
    architect: 'gpt-5.5',    // ← не существует
    tech_lead: 'glm',        // ← не model ID, а provider name
    researcher: 'gemini',    // ← не model ID, а provider name
    critic: 'gpt-5.5',       // ← не существует
    engineer: 'mimo',        // ← не model ID, а provider name
  };
  return map[roleId] ?? 'unknown';
}
```
Пишет мусор в `packet.modelTarget` (который валидируется роутером на непустоту, но больше нигде не читается).

**(3) `cfg.defaultModel` в адаптерах (хардкод) — ПОБЕЖДАЕТ.** Каждый адаптер в `send()` body:
`model: this.cfg.defaultModel` (`openai.adapter.ts:23`, `mimo.adapter.ts:23`, `glm.adapter.ts:23`,
`gemini.adapter.ts:30`). Именно это уходит в API request.

**Следствие:** сейчас реально уходит `cfg.defaultModel` (`gpt-4o`, `mimo-7b`, `glm-4-plus`, `gemini-1.5-pro`).
manifest.model + packet.modelTarget — мёртвые поля с мусорными значениями.

**Проблема для Wave 10:** реальные API требуют конкретных model IDs:
- MiMo принимает `mimo-v2.5-pro` (НЕ `mimo-7b` → 400 "Unsupported model").
- OpenRouter принимает `provider/model` формат (`anthropic/claude-3.5-sonnet`, НЕ `gpt-4o`).
→ Без починки model-routing Orchestra шлёт невалидные model → 404/400 от реальных API.

### 0.2. Что делает Phase 10-01

**Два изменения (дизайн-fix) + wiring ключей:**

1. **Починка designed data flow** — сделать YAML `model:` авторитетным источником (как и задумано
   архитектурой — `ContextPacket.modelTarget` существует, роутер валидирует):
   - (a) `RoundOrchestratorGatingAdapter.evaluate`: `buildPacket({..., model: manifest.model})` (+1 поле в вызове).
   - (b) `BuildPacketRequest`: +`model?: string` опциональное поле.
   - (c) `packet-builder.ts`: `modelTarget = req.model ?? resolveModelTarget(roleId)` — manifesto-driven
     с fallback на старую таблицу (для обратной совместимости; таблицу НЕ удаляем в этой фазе — deprecation).
   - (d) 4 адаптера `send()` body: `model: packet.modelTarget ?? this.cfg.defaultModel` (+fallback).

2. **Wiring реальных ключей** — `role-manifests/*.yaml` `model:` обновить на реальные model IDs,
   `apps/api/.env` (НЕ коммитить, создаёт owner/tech lead) с реальными ключами + base URLs.

### 0.3. Архитектурное решение: model — responsibility манифеста (как и provider)

**Выбор:** `manifest.model` (YAML) = авторитетный источник, а не env override и не хардкод адаптера.

**Обоснование:**
- Манифест уже несёт `provider:` (авторитетный — роутер его читает). `model:` — его естественный спутник:
  per-role конфигурация. Стратегическая модель (tech_lead → claude-3.5-sonnet), кодер (engineer → mimo-v2.5-pro),
  critic (→ claude-3.5-sonnet) — это **role-policy decisions**, живут в манифесте, как и provider.
- `ContextPacket.modelTarget` уже существует (Phase 3) и валидируется роутером (Phase 4) — дизайн это
  предусматривал. Packet-builder просто пишет из мусорного источника, адаптеры забыли читать.
- НЕ env override (как baseUrl): base URL — infra concern (меняется per-deployment: dev/prod/proxy),
  model — **policy concern** (меняется per-role: стратегия vs код). Разные уровни, разные источники.
- НЕ хардкод адаптера (как cfg.defaultModel): хардкод не выражает per-role выбор (один OpenAIAdapter
  обслуживает architect+critic, но им могут нужны разные модели).

**Fallback chain (после починки):** `packet.modelTarget` (из manifest.model) → `cfg.defaultModel` (хардкод адаптера).
Манифест побеждает, хардкод — safety net. Мёртвая `resolveModelTarget` таблица остаётся как deprecation
fallback (для ролей без `model:` в манифесте), убирается в cleanup-фазе.

## 5. Design notes

1. **manifest.model = authority, как и manifest.provider.** Манифест уже несёт provider (роутер читает).
   model — его спутник: per-role policy. Разные роли → разные модели (стратегия vs код vs research).
2. **НЕ env override для model (в отличие от baseUrl).** base URL = infra (per-deployment: dev/prod/proxy).
   model = policy (per-role). Разные уровни, разные источники. Манифест — правильное место для model.
3. **Fallback chain:** `packet.modelTarget` (manifest) → `cfg.defaultModel` (хардкод адаптера). Манифест
   побеждает, хардкод — safety net. resolveModelTarget таблица — deprecation fallback (для ролей без model).
4. **OpenRouter = OpenAI-compatible.** Идёт через OpenAIAdapter + `OPENAI_BASE_URL=openrouter.ai/api/v1`.
   Model format = `provider/model` (`anthropic/claude-3.5-sonnet`, `google/gemini-2.0-flash`). Adapter body
   `model: packet.modelTarget` прокидывает это как-is — OpenRouter принимает provider-qualified строку.
5. **MiMo = OpenAI-compatible.** MiMoAdapter + `MIMO_BASE_URL=token-plan-sgp.xiaomimimo.com/v1`.
   Model = `mimo-v2.5-pro` (НЕ `mimo-7b` хардкод — 400 Unsupported model). Verified probe 2026-07-23.
6. **GLM pending (D-10-1).** Ключ валиден (auth OK), баланс 0 (ошибка 1113). tech_lead временно на OpenRouter.
   Когда owner пополняет GLM — `tech_lead.yaml` provider→glm + `GLM_API_KEY`/`GLM_BASE_URL` в .env. 1 строка.
7. **Мёртвая resolveModelTarget НЕ удаляется (D-10-3).** Deprecation fallback для ролей без `model:`. Cleanup —
   отдельная фаза (не смешивать). Сейчас она перекрыта `req.model ??` — manifest побеждает.
8. **Safety net cfg.defaultModel.** Если manifest.model пустой/невалидный — адаптер использует хардкод.
   Не падает. Только реальный API может дать 400/404 (модель не существует) — это ловится в D-30.
9. **LLM_TIMEOUT_MS=60000 для real API.** Реальные модели медленнее mock stub-server (генерация ответа).
   60 сек — разумный default для real-provider setup (Wave 9 был 30s для mock).
10. **Сравнение с Wave 9:** 9-2 доказал LLM-цепочку через mock. 10-01 доказывает её на реальных провайдерах.
    Если 10-01 D-30/D-31 PASS → Orchestra production-ready с real LLM, north-star «оркестр оживает» полностью подтверждён.
